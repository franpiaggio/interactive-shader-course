Raymarching for Creative Art · Lesson 01 · Foundations

Every pixel runs your code

Before you can raymarch a single sphere, you need the one idea every shader is built on: your program runs once per pixel, and its only job is to pick a color.

Your win this lesson Write a fragment shader that turns each pixel's position into a color — and understand the uv coordinate, which is line 1 of every raymarcher you'll ever write.

The whole mental model

A fragment shader is a tiny program the GPU runs once for every pixel on the screen, all at the same time. There is no loop over pixels that you write — the GPU calls your function millions of times in parallel. Each call gets one input (which pixel am I?) and must produce one output (what color is it?).1

That's the entire contract. A 3D raymarched scene, a fractal, a cloud — all of it is just "given this pixel, compute a color." The art is in the computation.

Where am I? — fragCoord and uv

The GPU tells each invocation its pixel position in fragCoord, measured in pixels from the bottom-left. Raw pixels aren't portable (a 4K screen has different numbers than your phone), so the first thing every shader does is normalize them by the resolution:

vec2 uv = fragCoord / iResolution.xy;   // now 0.0 → 1.0 across the screen

Now uv.x runs 0 (left) → 1 (right) and uv.y runs 0 (bottom) → 1 (top), on any screen. uv is the per-pixel knob that drives the whole image. Feed it into a color and you can literally see the coordinate system:

Do this Move your mouse over the image. The bar reads the exact fragCoord, uv, and rgb under your cursor — live. Hover the bottom-left corner (uv ≈ 0,0, black), then the top-right (uv ≈ 1,1, yellow). You're literally reading the value a single GPU invocation computed for that pixel.

Black bottom-left (uv = 0,0), red-ish bottom-right (uv.x = 1), green-ish top-left (uv.y = 1), yellow top-right (both = 1). You're seeing the coordinate system painted directly onto the screen.

Reading the signature

void mainImage(out vec4 fragColor, in vec2 fragCoord) { ... }

This is the mainImage entry point — the function the harness (and Shadertoy) calls for every fragment.

  • in vec2 fragCoord — the input: this pixel's position. (in = given to you.)
  • out vec4 fragColor — the output: the color you must write. (out = you fill it in.)
  • A vec4 is four floats: (r, g, b, a), each 0→1. vec3 is just (r,g,b).

Peek ahead — drag, no code

Challenge 3 made a field that's 0 at a point and grows with distance. Give that field a center and a radius and you have a circle — and a circle's distance field is the 2D baby version of the sphere you'll raymarch. Drag the sliders: uCx/uCy move the center, uR sets the radius. No typing — just feel how the coordinate drives the shape.

This is the whole arc in miniature d = distance(uv, center) - radius is a signed distance field: negative inside, zero on the edge, positive outside. Swap distance in 2D for length in 3D and that single line becomes the sphere in the raymarch loop. That's Lessons 02–04.

Why this is the raymarching foundation

Here's the payoff. Open any raymarched scene on Shadertoy and the first line of real work is almost always this:

vec2 uv = (fragCoord - 0.5*iResolution.xy) / iResolution.y;  // centered, aspect-correct

Same idea you just learned — normalize the pixel — only centered on the screen (so 0,0 is the middle) and divided by height so circles stay round on wide screens. In a raymarcher, this uv becomes the direction we shoot a ray through that pixel into a 3D world. Master the humble coordinate now and the camera math later will feel obvious.
