Raymarching with Fragment Shaders · Lesson 04 · The leap to 3D

Raymarch your first sphere

This is the lesson the whole course points at. Same distance fields, one more coordinate — plus a loop that walks a ray through space until it hits something. A real 3D object, with no mesh anywhere.

Your win this lesson Render a 3D sphere by writing the sphere-tracing loop yourself — the single trick behind every raymarched scene on Shadertoy.

The idea in one paragraph

For each pixel, shoot a ray from the camera through that pixel into the scene. Walk forward along the ray until you're touching a surface. Whatever you touch tells you the pixel's color. The scene is still just an SDF — now map(vec3 p) instead of vec2:

float map(vec3 p) { return length(p) - 1.0; }   // a unit sphere — your Lesson 02 circle, in 3D

From pixel to ray — where the 3D comes from

This is the one genuinely new idea, so go slow. Picture a pinhole camera: your eye sits at a point, and a flat "sensor" floats one short distance in front of it. Each pixel is a spot on that sensor. The ray for a pixel is simply the line from the eye, through that spot, out into the scene.

You already have the spot: it's uv, the same centered, aspect-correct coordinate from Lesson 02 — now read as a position on the sensor (uv.x across, uv.y up). To make a 3D direction you just add the third axis — how far forward the sensor sits:

vec3 ro = vec3(0.0, 0.0, 3.0);          // the eye: 3 units back, looking toward -Z
vec3 rd = normalize(vec3(uv, -1.5));    // through pixel (uv) and 1.5 units forward (-Z)
Read the vector vec3(uv, -1.5) is (uv.x, uv.y, -1.5): sideways + up come from the pixel, and -1.5 is how far ahead the sensor sits (negative because the camera looks down −Z). That number is the focal length — your zoom. Bigger magnitude = rays bunch toward straight-ahead = narrower view = zoomed in. normalize makes it length 1 so that t below measures true distance. In Lesson 05 you'll replace this fixed eye with a camera you can orbit — but the idea is exactly this.

Sphere tracing — why the loop works

Here's the beautiful part. map(p) is the distance to the nearest surface. So from any point you can safely jump forward by exactly that distance without ever passing through a surface. Far away → big jumps (fast). Close to a surface → tiny jumps (precise). Step by the distance, every time:1

float t = 0.0;                          // distance travelled along the ray
for (int i = 0; i < 100; i++) {
    vec3 p = ro + rd * t;               // current point: origin + direction * distance
    float d = map(p);                   // distance to nearest surface
    if (d < 0.001) { hit = true; break; } // close enough → hit
    if (t > 50.0) break;                // gone too far → miss (sky)
    t += d;                             // THE step: advance by the safe distance
}

ro is the ray origin (camera). rd is the ray direction for this pixel — built from the uv you already know. That t += d line is the entire technique.

Play — feel the geometry

No code: radius resizes the sphere, camera z dollies the camera back and forth. Brighter = closer (shading by depth). Notice how a bigger sphere or a closer camera fills more of the frame — pure ray geometry.

Now you — complete the loop, graded live

The starter is missing one line — the safe step. Without it the ray never moves, so nothing is ever hit and you see only sky. Add it, then color the sphere. Check reads the pixels and grades you.

Watch for Loop counters are int; distances are float. Keep i < 100 an int compare and don't mix types inside the loop. Also: too few iterations makes grazing rays miss the surface (the silhouette frays) — 100 is plenty here.
